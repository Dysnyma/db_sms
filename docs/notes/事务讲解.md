# SQL 事务（Transaction）讲解

## 一句话解释

> 把多个操作打包成一个"整体"，要么全成功，要么全失败。不存在成功一半的情况。

---

## 银行转账的例子

```sql
-- A 给 B 转 100 块，两步操作：
UPDATE account SET balance = balance - 100 WHERE name = 'A';   -- 步骤①
UPDATE account SET balance = balance + 100 WHERE name = 'B';   -- 步骤②
```

如果步骤①成功、步骤②失败了 —— **100 块钱凭空消失了**。

事务就是解决这个问题：

```sql
START TRANSACTION;   -- 开始打包
  UPDATE account SET balance = balance - 100 WHERE name = 'A';
  UPDATE account SET balance = balance + 100 WHERE name = 'B';
COMMIT;              -- 都没问题 → 确认写入
```

如果中间出了错：

```sql
START TRANSACTION;
  UPDATE account SET balance = balance - 100 WHERE name = 'A';
  -- 这里出错了！
ROLLBACK;            -- 回滚 → 钱退回去，什么都没变
```

## 你的选课存储过程为什么需要事务

```sql
START TRANSACTION;

  SELECT ... INTO v_student_id ...    -- 步骤① 查学生
  各种校验 ...                         -- 步骤② 各种检查
  INSERT INTO enrollment ...          -- 步骤③ 插入选课记录
  -- 触发器自动更新 current_students   -- 步骤④ 触发器做的

COMMIT;  -- 四步都成功，才真正写入
```

如果有人同一秒选课，触发器更新人数时出错了 —— 选课记录也不应该留下。事务保证两者**原子**（绑在一起）。

## 四个核心概念（ACID）

| 字母 | 含义 | 大白话 |
|------|------|--------|
| **A** 原子性 | Atomicity | 要么全做，要么全不做 |
| **C** 一致性 | Consistency | 数据库从一个合法状态到另一个合法状态（库存不能为负） |
| **I** 隔离性 | Isolation | 多个人同时选课，互不干扰 |
| **D** 持久性 | Durability | 选课成功就是成功了，断电也不丢 |

## 常用语句

```sql
START TRANSACTION;   -- 开始事务
COMMIT;              -- 提交（确认写入）
ROLLBACK;            -- 回滚（撤销所有操作）
SAVEPOINT 点名;      -- 设置保存点
ROLLBACK TO 点名;    -- 回滚到某个保存点
```

---

## 自动提交与手动事务

```sql
-- MySQL 默认：每条语句自动提交（autocommit = ON）
UPDATE ...  -- 这条自动提交了

-- 显式事务：手动控制
START TRANSACTION;   -- 暂时关闭自动提交
  UPDATE ...
  UPDATE ...
COMMIT;              -- 恢复自动提交
```

## 错误处理（你代码里的那行）

```sql
DECLARE EXIT HANDLER FOR SQLEXCEPTION
BEGIN
    ROLLBACK;
    RESIGNAL;
END;
```

| 语句 | 含义 |
|------|------|
| `DECLARE EXIT HANDLER` | 声明一个"出错处理器" |
| `FOR SQLEXCEPTION` | 捕捉任何 SQL 错误 |
| `ROLLBACK` | 出错就回滚 |
| `RESIGNAL` | 把错误信息原样扔给调用者（不然静默失败） |

**放在存储过程最前面**，之后所有 SQL 都在事务保护下执行。
